# go expert review

## Cold-start findings

- **Go runtime init overhead is the dominant cost** — the Go scheduler, GC, and goroutine stack bootstrapping all fire before `main()`. On darwin/arm64 this accounts for most of the ~1.1 ms gap vs Zig (1.7 ms). This is not addressable without abandoning the Go runtime.
  - magnitude: 1-5 ms
  - confidence: definitely
  - cost: high (requires switching language)
  - patch sketch: not patchable in Go; accept as floor

- **`runtime.GOOS` check in `defaultEOL()` (line 163) is a runtime branch on a build-time constant** — Go does not eliminate this at compile time even though `runtime.GOOS` is a constant string. A `//go:build windows` tag with two files would make the dead branch unreachable and avoid the string comparison on every `run()` call (cheap but illustrates the pattern).
  - magnitude: sub-ms
  - confidence: probably
  - cost: low (two build-tag files, no logic change)
  - patch sketch: `eol_unix.go` returns `"\n"`, `eol_windows.go` returns `"\r\n"`, both guarded by `//go:build`

- **`fmt` import drags in printf machinery** — `fmt.Fprintf` and `fmt.Sprintf` are used at lines 86, 94, 104, 112, 117, 139. The format machinery adds ~150 KB to the binary and is initialized at startup. Replacing all `fmt.Fprintf(w, "die: %s\n", msg)` with `io.WriteString(w, "die: "+msg+"\n")` removes the `fmt` import entirely.
  - magnitude: sub-ms (less format-package init work)
  - confidence: probably
  - cost: low
  - patch sketch: replace 6 `fmt.Fprintf/Sprintf` call sites with string concat + `io.WriteString`

## Binary-size findings

- **`fmt` package is the largest removable chunk** — `fmt` pulls in `reflect`, `strconv`, and printf infrastructure. Estimated ~200 KB of the 1.6 MB binary. Current usage is only for error message formatting (lines 86, 94, 104, 112, 117, 139) — all are trivial `"die: " + literal + "\n"` patterns that need no format verbs.
  - magnitude: medium (5-20%, ~200 KB / 12%)
  - confidence: definitely
  - cost: low
  - patch sketch: remove `fmt` import; use `io.WriteString(w, "die: "+msg+"\n")` everywhere

- **`strings` package carries substantial bulk** — used for `strings.Join`, `strings.HasPrefix`, `strings.TrimPrefix`, `strings.TrimFunc`. The `strings` package itself is modest but links against `unicode` tables for `TrimFunc`. Since `trimASCII` (line 198) only needs 6 ASCII bytes, a hand-rolled byte-scan loop replaces `strings.TrimFunc` + `isASCIIWhitespace` and avoids the unicode dependency.
  - magnitude: small (<5%, ~30-60 KB)
  - confidence: probably
  - cost: low
  - patch sketch: replace `strings.TrimFunc(s, isASCIIWhitespace)` with a manual `for i` loop over bytes

- **Go runtime GC and goroutine infrastructure is always present** — reflects in the 1.6 MB floor. Cannot be trimmed further with standard `gc` toolchain. `tinygo` can reduce to ~100 KB but breaks stdlib compatibility (`os`, `io.ReadAll`, etc.) and is not production-ready for this use case.
  - magnitude: large (>20%)
  - confidence: definitely (floor is fixed)
  - cost: high (requires different toolchain)
  - patch sketch: not recommended; accept Go's floor or switch language

- **`-ldflags "-s -w"` already applied** — DWARF and symbol table are stripped. No further linker-flag gains available with standard toolchain. `garble` obfuscation would not help size here.
  - magnitude: none remaining
  - confidence: definitely
  - cost: n/a

## Code quality

- **`defaultEOL()` called unconditionally on every `run()` entry (line 53) even when `--eol` is explicitly set** — the result is then overwritten by `parseEOL` (lines 102, 109). Minor allocation concern but more importantly signals the build-time-constant issue above.

- **`joinArgs` has a dead `default` branch (line 195)** — `parseTrim` only accepts `"each"`, `"all"`, `"none"` (lines 170-175) and the default trim is `"each"`. The `default: return strings.Join(args, sep)` at line 195 is unreachable. Dead code bloats instruction cache marginally and confuses readers.

- **`appendEOL` (line 213) allocates a new string via `s + eol` on every ARG-path call** — for the common case (output already ends with `\n`), `endsWithEOL` short-circuits and no alloc occurs. But when appending, the concatenation forces a heap copy. Using `io.WriteString` twice (body then eol separately) on the writer would avoid this alloc entirely and remove the `appendEOL` helper.

- **`isTTY` type-assertion to `*os.File` (line 249) is sound** but `f.Stat()` issues a `fstat` syscall. On the ARG path (`sawDashDash == true`) `isTTY` is never called — this is correct. No issue, just noting the syscall is bounded to stdin path only.

## Risks / blockers

- Removing `fmt` must be verified against all error paths in `tests/run.sh` — the error message format `"die: %s\n"` must be preserved exactly.
- Build-tag split for EOL requires CI to test `GOOS=windows` cross-compile (currently not in matrix per findings doc).
- `tinygo` is not a realistic option: `io.ReadAll` and `os.Stdin` behave differently.

## TL;DR

- **Binary size**: Remove `fmt` import (replace 6 call sites with string concat) — ~200 KB / 12% reduction, low cost, definitely effective.
- **Cold-start**: Replace `runtime.GOOS` runtime branch with `//go:build` tag files — sub-ms gain, low cost; the 1.1 ms gap vs Zig is the Go runtime floor and cannot be closed without changing language.
- **Code quality**: Eliminate unreachable `default` branch in `joinArgs` (line 195) and avoid the `s + eol` heap alloc in `appendEOL` by writing body and terminator separately to the writer.
