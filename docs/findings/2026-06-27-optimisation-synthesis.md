# Cross-language optimisation review (synthesis)

Date: 2026-06-27. Generated by die-impl-expert-review-v2 workflow.

## Top picks (ROI-ranked, language-agnostic)

- Rank 1: **Go** — Remove `fmt` import (~200 KB, 12% binary shrink; low cost). Replace 6 `fmt.Fprintf/Sprintf` call sites with `io.WriteString` + string concat. Highest absolute size impact of any single change across all four languages, zero behaviour risk, verified by grep of existing call sites.

- Rank 2: **Rust** — Replace `writeln!` format machinery with chained `write_all` byte-slice calls (estimated 5–15% / ~15–44 KB; low cost). Removes `core::fmt` dispatch vtable pulled in by two `writeln!` calls. Easy patch, safe to test against existing stderr message checks.

- Rank 3: **MoonBit** — Add `strip -x` post-build step in justfile (medium impact, ~10–20 KB; low cost, one line). Residual debug sections are definitely present; this is a build-system change only, no source changes.

- Rank 4: **Zig** — Remove `defer arena.deinit()` / `defer buf.deinit()` (sub-ms cold-start gain; low cost). OS reclaims memory on exit; these free() calls are pure overhead on every run. Also add `comptime` qualifiers to `resolveEol` and `_setmode` OS branches to guarantee dead-code elimination (no runtime cost, improves readability).

ROI drops below "worth doing" after Rank 4; remaining candidates are either high-cost rewrites or sub-1% gains.

## Per-language verdict

- **go**: Do remove the `fmt` import (Rank 1 above) and the unreachable `default` branch in `joinArgs`; also switch `defaultEOL` to `//go:build` tag files (low cost, cleaner intent). Defer the `appendEOL` heap-alloc fix (requires writer refactor, marginal gain). Drop tinygo — stdlib incompatibility makes it a non-starter. The 1.1 ms cold-start gap vs Zig is a Go runtime floor; accept it.

- **rust**: Do replace `writeln!` with `write_all` byte slices (Rank 2). Also do the `Vec::drain` refactor at the `--` split (zero-cost, definitely faster on ARG path). Defer `args_os()` migration until non-UTF-8 argument behaviour is specced. Drop `#![no_std]` — not viable without a full I/O rewrite; the stdlib carry-along is an accepted cost.

- **mbt**: Do add `strip -x` post-build immediately (Rank 3, one justfile line). Do cache `is_windows_host()` result to eliminate the redundant `getenv` call (low cost). Do fix the dead branch in `append_eol_str` and switch `read_stdin_all` to bulk copy. Defer dropping `moonbitlang/core/env` (B1) — high impact but high risk; the stub.c rewrite is non-trivial and correctness risk on edge-case argv is real.

- **zig**: Do remove the `defer deinit()` calls (Rank 4) and add `comptime` to OS branches. Do replace `ArenaAllocator+page_allocator` with `FixedBufferAllocator` on a stack buffer (covers both cold-start and binary-size axes, low cost). Defer the `buildArgOutput` stack fast-path — worthwhile but more invasive. The 50 KB darwin binary is already near-optimal; no structural size wins remain.

## What I would NOT touch

- **Go: tinygo** — stdlib incompatibility (`io.ReadAll`, `os.Stdin` behave differently); not production-ready for this use case.
- **Rust: `#![no_std]`** — would require reimplementing all I/O via `extern "C"`; Windows support would need extensive work. The 296 KB stdlib floor is the accepted tradeoff.
- **MoonBit: dropping `moonbitlang/core/env` (B1)** — highest-impact size change but the correctness risk on argv edge cases (null bytes, multi-byte paths) and the undocumented `moon build` Clang flag pass-through make it a deferred item, not a "now" item.
- **Zig: musl static linking** — inflates darwin binary from 50 KB to 61 KB; the linux-gnu 11 KB path is already optimal and the darwin gap is structural dylib overhead, not fixable by flag.
- **MoonBit: LLVM IR backend** — not yet surfaced in `moon` CLI; no actionable path today.
